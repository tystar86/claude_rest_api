# GitHub Actions Production Release

This workflow gives Blogit a production release path without copying source code to the VPS.

The release shape is:

- GitHub Actions runs the release manually
- backend and frontend tests pass first
- Docker images are built and pushed to private GHCR
- the VPS receives the tracked [docker-compose.production.yml](../../docker-compose.production.yml) and a generated `.release.env`
- the VPS pulls images and restarts containers with `docker compose`

## What Stays On The VPS

The server keeps only runtime state:

- `.env.production`
- `docker-compose.production.yml`
- `.release.env` generated by the workflow
- Docker volumes such as Postgres data
- Caddy and its config

The repo itself does **not** need to be cloned on the server.

## Required GitHub Actions Secrets

Set these repository secrets before running the workflow:

- `GHCR_PULL_TOKEN` — read-only token used on the VPS during deploy
- `VPS_HOST` — production server hostname or IP
- `VPS_USER` — deploy user on the VPS
- `VPS_SSH_KEY` — private SSH key for that deploy user
- `VPS_PORT` — optional SSH port; defaults to `22` when unset
- `VPS_DEPLOY_PATH` — deployment directory, for example `/srv/blogit`

## Required VPS State

Before the first release:

1. Create the deployment directory:

```bash
mkdir -p /srv/blogit
```

2. Create the runtime env file from your local checkout:

```bash
scp .env.example deploy@your_server_ip:/srv/blogit/.env.production
ssh deploy@your_server_ip
nano /srv/blogit/.env.production
```

3. Fill in production secrets and host settings in `.env.production`.

4. Verify Docker and the Compose plugin are installed on the VPS.

5. Verify the shared `edge` Docker network already exists if Caddy is running outside this stack.

The workflow will upload `docker-compose.production.yml` and `.release.env` for each release, so those files do not need to be managed manually after the initial setup.

GitHub Actions pushes the images with the built-in `GITHUB_TOKEN`, so you do not need a separate GHCR push secret for this repository.

## How The Workflow Tags Images

Each production release publishes:

- `ghcr.io/<owner>/blogit-backend:sha-<12-char-commit>`
- `ghcr.io/<owner>/blogit-frontend:sha-<12-char-commit>`
- `ghcr.io/<owner>/blogit-backend:latest`
- `ghcr.io/<owner>/blogit-frontend:latest`

The deploy step uses the immutable `sha-...` tag and writes it into `.release.env` on the server.

## Running A Release

1. Open GitHub Actions.
2. Run `Production release`.
3. Make sure the selected ref is `master`.
4. Wait for the workflow to finish the test, build, push, and deploy jobs.

The workflow will:

- fail if the selected ref is not `master`
- update the compose file on the VPS
- write the current image tag to `.release.env`
- pull the tagged images from GHCR
- restart the Blogit services in place

## Operational Notes

- Local development still uses [docker-compose.local.yml](../../docker-compose.local.yml).
- Production now uses registry images only; it does not build on the VPS.
- If you later add staging, the same image tags can be promoted there before production.
- Robot and security workflows remain separate today; this release workflow currently gates on backend and frontend CI only.
- If GHCR packages were first created manually with a PAT and are not linked to this repository, the first `GITHUB_TOKEN` publish may need a one-time package permission/linking fix in GitHub.
